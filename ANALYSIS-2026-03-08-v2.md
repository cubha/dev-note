## 1. 핵심 문제 재정의

### 1.1 왜 두 문제를 함께 풀어야 하는가

```
Smart Paste 출력 포맷  ←──── 의존 ────→  데이터 모델
       │                                      │
       ▼                                      ▼
 AI JSON Schema            ←── 동일 구조 ──→  content 필드 포맷
       │                                      │
       ▼                                      ▼
 SmartPastePanel UI         ←── 렌더링 ──→   CardDetailEditor UI
```

- Smart Paste의 AI 출력 스키마는 저장 포맷(content)에 종속
- 현재 포맷(StructuredContent)에 맞춰 Smart Paste를 개선하면 → IA 변경 시 전면 재작성
- **올바른 순서**: 저장 포맷 설계 → AI 스키마 설계 → UI 렌더러 → Smart Paste 통합

### 1.2 현재 구조의 정확한 한계

**데이터 모델 한계**:
```typescript
// 현재: 5종 고정 타입 × 고정 필드 스키마
type ItemType = 'server' | 'db' | 'api' | 'note' | 'custom'

// FIELD_SCHEMAS는 컴파일 타임에 고정 (types.ts:42-73)
// server: [host, port, username, password, keyPath, note]
// db:     [host, port, dbName, username, password, note]
// api:    [url, method, apiKey, token, headers, note]
// note:   [content]
// custom: [content]
```

문제:
1. **혼합 불가**: 하나의 카드에 서버 + DB + URL + 메모를 함께 저장할 수 없음
2. **스키마 경직**: 필드 추가/삭제가 코드 변경 없이 불가능
3. **관계 단절**: 같은 프로젝트의 관련 정보들이 개별 카드로 분절

**Smart Paste 한계**:
```
현재: 접속 정보 패턴 파싱기
  - Tier 1: SSH config, DB URI, curl 등 정형 패턴 매칭
  - Tier 2: Claude API → 동일한 필드 추출 (host, port 등)
  → 5종 고정 타입의 고정 필드에 매핑하는 것이 목적

필요: 자유형 텍스트 → 의미 단위 섹션 구조화 엔진
  - 맥락을 이해하고 관련 정보를 그룹핑
  - 접속 정보/URL/메모 등을 자동 분류
  - 하나의 문서 안에 다양한 섹션으로 조직화
```

## 2. 레퍼런스 아키텍처 분석

### 2.1 Notion의 블록 데이터 모델

> 출처: [The data model behind Notion](https://www.notion.com/blog/data-model-behind-notion)

```
Block {
  id: UUID
  type: string          // 렌더링 방식 결정
  properties: {}        // 타입별 데이터
  content: BlockID[]    // 자식 블록 ID 배열 (순서 유지)
  parent: BlockID       // 부모 블록 포인터
}
```

**핵심 원칙**: "Everything is a block" — 페이지도 블록, 텍스트도 블록, 데이터베이스 행도 블록
**로컬 캐시**: IndexedDB/SQLite 기반 RecordCache (LRU) + TransactionQueue

**dev-note 적용 시 과도한 부분**:
- 중첩 블록 트리 → dev-note의 접속 정보에는 불필요한 복잡도
- 실시간 협업/CRDT → 로컬 전용 앱에 불필요
- 블록별 권한 상속 → 단일 사용자 앱에 불필요

### 2.2 AFFiNE의 로컬-퍼스트 블록 에디터

> 출처: [AFFiNE vs AppFlowy vs Anytype](https://affine.pro/blog/affine-vs-appflowy-vs-anytype)

- BlockSuite + CRDT(y-octo) 기반
- 로컬 저장 우선 + 선택적 클라우드 동기화
- 블록: rich text, sticky note, embed, database view, shape

**dev-note 시사점**: 로컬-퍼스트 철학은 동일하나, CRDT/협업 인프라는 과도

### 2.3 블록 에디터 라이브러리 비교

> 출처: [Liveblocks Blog](https://liveblocks.io/blog/which-rich-text-editor-framework-should-you-choose-in-2025), [BlockNote vs Tiptap](https://tiptap.dev/alternatives/blocknote-vs-tiptap)

| 라이브러리 | 기반 | 번들 크기 | 블록 에디터 | React 네이티브 | 접속 정보 렌더링 |
|---|---|---|---|---|---|
| **TipTap** | ProseMirror | ~200KB | 확장으로 구현 | ✅ | 커스텀 노드로 가능 |
| **BlockNote** | TipTap+ProseMirror | ~350KB | 네이티브 | React 전용 | 커스텀 블록으로 가능 |
| **Plate.js** | Slate | ~250KB | 플러그인 | ✅ | 커스텀 엘리먼트 |
| **CodeMirror 6** (현재) | 자체 | ~180KB | ❌ 텍스트 전용 | ✅ | 위젯/데코레이션 제한적 |

**결론**: 블록 에디터 도입 여부가 핵심 분기점

## 3. 설계 방향 의사결정

### 3.1 블록 에디터 도입 여부 — **도입 불필요 (권장)**

**이유**:
1. **번들 크기**: BlockNote(350KB) + 기존 CodeMirror(180KB) = 530KB 추가 vs 현재 전체 앱 ~1.7MB
2. **학습 곡선**: ProseMirror 기반 에디터의 커스텀 노드/스키마 학습 비용
3. **핵심 사용 패턴**: SI 프리랜서가 접속 정보를 "편집"하는 빈도 < "조회/복사"하는 빈도
4. **CLAUDE.md 규칙**: 기술 스택 고정값 변경 금지 — CodeMirror 6 유지
5. **목적 부합도**: 접속 정보 관리 도구에 Notion 수준 블록 에디터는 과도

**대안 접근**: **섹션 기반 커스텀 렌더러** (블록 에디터 없이)
- React 컴포넌트로 직접 섹션 렌더링
- CodeMirror는 마크다운/텍스트 편집에만 사용 (현재와 동일)
- 구조화 데이터(접속 정보, URL)는 전용 폼 컴포넌트로 편집

### 3.2 점진 전환 전략 — Phase B 채택 (수정안)

이전 보고서의 Phase B(HybridContent)를 기반으로 하되, **블록 에디터 없이** 구현:

```
[AS-IS]
Folder → Item { type: 5종, content: StructuredContent }
  → 카드 1개 = 접속 정보 1개

[TO-BE: Phase B']
Folder → Item { type: 6종(+document), content: StructuredContent | HybridContent }
  → 기존 5종 카드 100% 호환
  → document 타입: 여러 섹션을 하나의 문서로 관리
  → AI가 자유 텍스트 → Section[] 변환
```

## 4. 상세 데이터 모델 설계

### 4.1 HybridContent 최종 설계

```typescript
// ─── 새로운 콘텐츠 포맷 (기존 StructuredContent/LegacyContent 공존) ──

/** 하이브리드 콘텐츠 — document 타입 전용 */
interface HybridContent {
  format: 'hybrid'
  sections: Section[]
}

/** 섹션: 문서 내 의미 단위 블록 */
interface Section {
  id: string              // nanoid(12)
  type: SectionType
  title: string           // 섹션 제목 (빈 문자열 허용)
  collapsed: boolean      // 접힘/펼침 상태
}

type SectionType = 'markdown' | 'credentials' | 'urls' | 'env' | 'code'

// ─── 섹션 타입별 데이터 ────────────────────

/** 마크다운 텍스트 섹션 */
interface MarkdownSection extends Section {
  type: 'markdown'
  text: string
}

/** 접속 정보 섹션 (서버/DB 통합) */
interface CredentialSection extends Section {
  type: 'credentials'
  items: CredentialEntry[]
}

interface CredentialEntry {
  id: string              // nanoid(8)
  label: string           // "운영서버", "개발 DB", "Redis"
  category: 'server' | 'database' | 'other'
  host: string
  port: string            // 빈 문자열 허용
  username: string
  password: string        // 저장 시 평문, 렌더링 시 마스킹
  database?: string       // DB 전용
  extra: string           // 비고/메모 (자유 텍스트)
}

/** URL 모음 섹션 */
interface UrlSection extends Section {
  type: 'urls'
  items: UrlEntry[]
}

interface UrlEntry {
  id: string
  label: string           // "관리자 페이지", "API 문서"
  url: string
  method?: string         // API 전용 (GET/POST 등)
  auth?: string           // 인증 정보 메모
  note: string
}

/** 환경변수 / Key-Value 섹션 */
interface EnvSection extends Section {
  type: 'env'
  pairs: EnvEntry[]
}

interface EnvEntry {
  key: string
  value: string
  secret: boolean         // true → 마스킹 렌더링
}

/** 코드 스니펫 섹션 */
interface CodeSection extends Section {
  type: 'code'
  language: string        // 'bash', 'sql', 'json' 등
  code: string
}

// ─── 통합 타입 ──────────────────────────────

type AnySection = MarkdownSection | CredentialSection | UrlSection | EnvSection | CodeSection

type CardContent = StructuredContent | LegacyContent | HybridContent
```

## 5. Smart Paste AI 재설계

### 5.1 새로운 2-Tier 아키텍처

```
[Tier 1: 정규식 사전 분석 — 역할 변경]
  기존: 5종 타입 중 하나로 매핑 → 고정 필드 추출
  변경: 텍스트 내 패턴 감지 → AI 프롬프트 힌트 생성

[Tier 2: Claude API — 역할 확장]
  기존: 동일한 필드 추출 (host, port 등) → SmartPasteResult
  변경: 의미 단위 섹션 분류 + 구조화 → Section[]
```

### 5.2 새로운 흐름

```
사용자 입력 (자유형 텍스트)
  │
  ▼
[Tier 1: 패턴 힌트 생성]
  - detectPatterns(text) → PatternHint[]
  - 예: [{type:'credential', match:'10.0.0.1:22'}, {type:'url', match:'https://...'}]
  - 목적: AI 프롬프트에 힌트를 제공하여 정확도 향상 + 토큰 절약
  │
  ▼
[분기: AI 키 유무]
  │
  ├── AI 키 있음 → Tier 2 AI 구조화
  │     - 입력: 원본 텍스트 + 패턴 힌트
  │     - 출력: Section[] (HybridContent.sections)
  │     - Claude Structured Outputs로 스키마 강제
  │
  └── AI 키 없음 → Tier 1 폴백 (기존 방식)
        - 접속 정보 패턴 → CredentialSection 자동 생성
        - URL 패턴 → UrlSection 자동 생성
        - 나머지 → MarkdownSection으로 저장
        - 완벽하진 않지만 기본 구조화 가능
```

### 5.3 AI JSON Schema (새로운 DOCUMENT_PASTE_SCHEMA)

```typescript
export const DOCUMENT_PASTE_SCHEMA = {
  type: 'object' as const,
  properties: {
    title: {
      type: 'string' as const,

    sections: {
      type: 'array' as const,
      items: {
        type: 'object' as const,
        properties: {
          type: {
            type: 'string' as const,
            enum: ['markdown', 'credentials', 'urls', 'env', 'code'],
          },
          title: { type: 'string' as const },
          // 타입별 데이터는 아래 참조
          text: { type: 'string' as const },           // markdown
          items: { type: 'array' as const, items: { type: 'object' as const } }, // credentials, urls
          pairs: { type: 'array' as const, items: { type: 'object' as const } }, // env
          code: { type: 'string' as const },            // code
          language: { type: 'string' as const },        // code
        },
        required: ['type', 'title'] as const,
      },
    },
    confidence: {
      type: 'string' as const,
      enum: ['high', 'medium', 'low'],
    },
  },
  required: ['title', 'sections', 'suggestedTags', 'confidence'] as const,
  additionalProperties: false,
}
```

### 5.4 AI 프롬프트 설계

```
You are a content organizer for a developer's project information management tool.

Given free-form text (potentially with regex pattern hints), organize it into
meaningful sections for a structured document.

Section types:
- "credentials": Server/DB access info (host, port, username, password)
- "urls": Related URLs with labels
- "env": Environment variables or key-value configurations
- "code": Code snippets, commands, or scripts
- "markdown": Free-form notes, instructions, or documentation

Rules:
- Group related information into logical sections with descriptive Korean titles
- Detect credentials (host, port, username, password, database) and structure them
- Each credential entry needs: label, category("server"|"database"|"other"), host,
  port, username, password, database(if applicable), extra(notes)
- Detect URLs and categorize them with labels
- Detect key=value patterns as env sections (mark passwords as secret:true)
- Keep narrative text as markdown sections
- Preserve ALL original information — do not omit anything
- Generate section titles in Korean (예: "운영 환경 접속 정보", "관련 URL")
- If pattern hints are provided, use them to improve accuracy

Pattern hints format: [{"type":"credential","match":"10.0.0.1:22"},...]
```

### 5.5 기존 Smart Paste와의 호환성

```
[기존 5종 카드 생성 시]
  → CardFormModal → SmartPastePanel → 기존 localSmartParse + AIService.smartPaste
  → StructuredContent 출력
  → 변경 없음 (100% 호환)

[document 타입 생성 시]
  → 새로운 DocumentFormModal or 기존 CardFormModal 확장
  → 새로운 DocumentSmartPaste 컴포넌트
  → documentSmartPaste(text) → HybridContent.sections
  → HybridContent 출력
```

## 7. 구현 영향도 분석

### 7.1 변경 파일 목록

| 파일 | 변경 유형 | 변경 내용 |
|---|---|---|
| `core/types.ts` | **확장** | HybridContent, Section, CredentialEntry 등 타입 추가 |
| `core/db.ts` | **확장** | ItemType에 'document' 추가, v10 마이그레이션 |
| `core/content.ts` | **확장** | parseContent에 HybridContent 분기 추가 |
| `core/ai-schemas.ts` | **확장** | DOCUMENT_PASTE_SCHEMA 추가 |
| `core/ai.ts` | **확장** | AIService.documentSmartPaste() 메서드 추가 |
| `core/smart-paste.ts` | **확장** | detectPatterns() 힌트 생성 함수 추가 |
| `features/cards/DocumentEditor.tsx` | **신규** | document 전용 에디터 (섹션 리스트) |
| `features/cards/sections/` | **신규** | 섹션별 뷰/편집 컴포넌트 5종 |
| `features/cards/DocumentSmartPaste.tsx` | **신규** | document 전용 Smart Paste 패널 |
| `features/cards/CardDetailEditor.tsx` | **수정** | document 타입 분기 → DocumentEditor로 위임 |
| `features/cards/InfoCard.tsx` | **수정** | document 타입 카드 미리보기 |
| `features/cards/CardFormModal.tsx` | **수정** | document 타입 추가 |
| `features/dashboard/CardGrid.tsx` | **수정** | document 타입 필터 지원 |
| `store/atoms.ts` | **수정** | 필요 시 document 관련 atom 추가 |

### 7.2 변경하지 않는 파일 (호환성 보장)

- `features/sidebar/` — 폴더 트리 (type 무관)
- `features/storage/` — 내보내기/가져오기 (content는 JSON string이므로 자동 호환)
- `features/settings/` — 설정 모달 (변경 없음)
- `store/tabHelpers.ts` — 탭 관리 (변경 없음)
- `shared/` — 공통 컴포넌트 (변경 없음)

### 7.3 위험도 평가

| 리스크 | 심각도 | 완화 방안 |
|---|---|---|
| 기존 카드 데이터 손상 | 🔴 | content.ts 파싱 시 기존 format 우선 처리, v10 마이그레이션에 데이터 변경 없음 |
| 번들 크기 증가 | 🟡 | 새 컴포넌트 추가만 (~10KB), 라이브러리 추가 없음 |
| AI 프롬프트 비용 | 🟡 | Tier 1 힌트로 토큰 절약, haiku 모델 유지 |
| 내보내기/가져오기 호환 | 🟢 | content는 JSON string → HybridContent도 자동 직렬화/역직렬화 |
| 시맨틱 검색 호환 | 🟢 | extractSearchText 확장만 필요 (HybridContent → 평문 추출) |
